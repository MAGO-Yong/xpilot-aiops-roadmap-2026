# 🚀 XPILOT 告警智能诊断 — 2026年产品规划（V2）

## 一、建设背景

### 1.1 现状与瓶颈

当前故障应急体系依赖核心指标告警关联RCA进行诊断，存在明显局限：

**覆盖不足**：仅能处理预设的业务场景告警，无法有效应对日常生产中海量的、多样的告警。研发同学仍需投入大量精力进行人工排障。

**成本高昂**：每次故障排查需要跨指标、链路、日志等多维度人工查询，重复执行相似操作，平均耗时30-45分钟才能定位根因。

**诉求多样**：不同业务线有特定排障流程——电商业务查订单、推荐业务查召回、支付业务查渠道，现有平台无法灵活支撑业务定制化需求。

**核心矛盾**：基础设施诊断（CPU/RPC/日志）与业务指标诊断（CTR/订单/支付成功率）是两种完全不同的逻辑——前者通用可标准化，后者业务特异需定制。

---

### 1.2 核心场景与案例

#### 场景：P0/P1告警智能诊断

**场景描述**：核心业务P0/P1告警触发，需要在10分钟内完成诊断并给出修复建议。

**完整处理流程**：

```
告警触发
    ↓
[诊断执行层] 智能降噪评估
    - 判断是否为重复告警/已恢复/低优先级
    - 如果是：自动关闭/降优先级
    - 如果否：进入诊断流程
    ↓
[诊断执行层] 多Agent并行诊断（2-3分钟）
    ├─ Metrics Agent分析指标异常
    ├─ Logs Agent分析日志错误
    ├─ Change Agent关联最近变更
    └─ Topology Agent分析服务拓扑
    ↓
[AI决策引擎] 根因综合判定
    - 汇总各Agent诊断结果
    - 生成根因分析报告
    - 推荐修复方案
    ↓
输出：结构化诊断报告
    ├─ 根因判定（准确率目标75%+）
    ├─ 影响面分析
    ├─ 修复建议（回滚/重启/扩容/配置修复）
    └─ 相关上下文（指标图、日志、变更链接）
    ↓
[诊断执行层] 智能处置（根据风险等级）
    ├─ 低风险（重启、扩容）：自动执行
    ├─ 高风险（回滚）：人工确认后执行
    └─ 复杂场景：推荐SRE介入
    ↓
[诊断执行层] 复盘与优化
    - 自动生成事件时间线
    - 沉淀诊断知识
    - 优化Agent模型
```

### 1.4 MTTR 深度分析

#### 四个阶段的时间分布

```
告警触发（MTTD: 1-2min，占比5%）
    ↓
分类/确认（MTTA: 5-8min，占比15%）
    ↓
诊断/根因定位（MTTI: 30-45min，占比70%）【⚡ 最大瓶颈】
    ↓
修复/验证（MTTR: 5-10min，占比10%）
```

**💡 核心洞察**：诊断阶段占70%时间，是AI压缩的最大杠杆点。根据Metoro深度分析，大多数AI RCA工具失败的根本原因在于**数据基础问题**，而非算法本身。

| 维度 | 现状 | AI方案 | 预期效果 |

### 1.5 核心矛盾

基础设施诊断（CPU/RPC/日志）与业务指标诊断（CTR/订单/支付）本质上是**两种完全不同的诊断逻辑**：

| 类型 | 特点 | 建设方式 |
|------|------|---------|
| **基础设施问题** | 通用性强，可以标准化 | 平台统一建设，开箱即用 |
| **业务指标问题** | 业务特异，需要业务知识 | SRE+业务协同孵化，逐步沉淀 |

---

### 1.6 解决思路

**核心理念**：将人工排障经验转化为机器可执行的流程，构建**"智能发现→深度分析→智能处置"**自动化闭环。

**双轨制架构设计**：

| 轨道 | 定位 | 内容 |
|------|------|------|
| **🏗️ 平台侧：通用能力沉淀** | 基础设施诊断 | CPU/内存/RPC/日志标准化诊断，开箱即用，全业务自动继承 |
| **💼 业务侧：领域知识编排** | 业务指标诊断 | 电商/推荐/支付等业务SOP固化，SRE+业务协同孵化 |

**ReAct 技术内核**：Reasoning（推理）+ Acting（行动）循环，让AI Agent像专家一样动态选择工具、调整策略，而非执行固定流程。

## 五、技术方案

### 5.1 诊断场景层

#### 5.1.1 场景接入设计

诊断场景层作为统一入口，支持多场景标准化接入：

| 场景类型 | 触发条件 | 处理特点 |
|---------|---------|---------|
| **告警场景** | P0/P1告警触发 | 10分钟内响应，紧急处理流程 |
| **RCA场景** | 故障复盘需求 | 深度分析，历史数据关联 |
| **日常场景** | 主动巡检发现 | 预防性诊断，趋势分析 |
| **发布场景** | 变更发布期间 | 变更关联分析，快速回滚 |

#### 5.1.2 场景标准化流程

```
场景触发
    ↓
场景识别（告警/RCA/日常/发布）
    ↓
上下文富化（自动关联相关数据）
    ↓
进入诊断执行层
```

---

### 5.2 诊断执行层

诊断执行层是智能诊断的核心，包含**四个关键阶段**：整体定界→深度诊断→智能处置→复盘优化。

#### 5.2.1 整体定界

**目标**：快速理解问题上下文，判断问题性质，为后续诊断确定方向。

**【阶段一】风险感知**

将海量告警作为广泛的风险感知源，进行智能分类、收敛和路由：

| 功能 | 技术实现 | 预期效果 |
|------|---------|---------|
| 告警去重 | 基于指纹的重复告警识别 | 减少50%+重复告警 |
| 告警关联 | 基于拓扑关系的告警聚合 | 多告警合并为1个Incident |
| 动态基线 | 基于历史数据的动态阈值 | 减少30%+误报 |
| 告警分级 | AI评估业务影响面 | 自动标记P0/P1/P2 |
| 上下文富化 | 关联变更/相关服务/历史案例 | 为诊断提供上下文 |

**【阶段二】问题定性**

对路由来的告警，自动分析关联服务的系统、应用及业务指标，快速判断问题性质：

| 判断维度 | 分析内容 | 输出 |
|---------|---------|------|
| 系统层 | CPU、内存、磁盘、网络 | 资源瓶颈/正常 |
| 应用层 | QPS、延迟、错误率 | 时延/错误/正常 |
| 业务层 | CTR、订单量、支付成功率 | 业务跌/正常 |

**整体定界输出**：问题类型标签（时延类/错误类/资源类/业务类）+ 影响范围评估

---

#### 5.2.2 深度诊断（核心章节）

深度诊断采用**【特征→上下文→推理→决策】四阶段**流程，结合多Agent并行分析，实现像专家一样的根因定位。

**【阶段一】特征提取**

基于传统时序异常算法，对服务进行系统层、应用层、异常进行加工，提取各层服务风险特征。

**数据采集（"抽血化验"）**：

| 数据层 | 采集内容 | 用途 |
|--------|---------|------|
| **系统层指标** | CPU使用率、内存占用、磁盘IO、网络带宽 | 资源瓶颈检测 |
| **运行层指标** | JVM状态（堆内存、GC次数/耗时、线程数）、异常堆栈 | 运行时问题定位 |
| **接口层指标** | RPC成功率、HTTP延迟、QPS、错误码分布 | 接口健康度评估 |
| **链路数据** | 调用链、Span层级、调用耗时、状态码 | 依赖关系分析 |
| **日志数据** | 业务日志、框架日志、错误日志 | 异常模式识别 |
| **变更数据** | 代码发布、配置变更、扩缩容 | 变更关联分析 |

**时序异常检测**：
- 动态基线建立
- 异常波动标记
- 基线偏离分析

**【阶段二】上下文构建**

基于服务特征数据，通过专家经验&分级提示词，进行服务上下文构建。

**上下文要素**：
- 服务基础信息（服务名、部署信息、负责人）
- 历史健康状态（过去7天/30天趋势）
- 关联依赖拓扑（上下游服务清单）
- 最近变更记录（代码/配置/基础设施）
- 历史相似故障案例

**【阶段三】推理分析**

通过诊断Agent，结合服务风险特征、服务上下文及变更，进行服务分析，问题定性。

**多Agent并行诊断**：

| Agent | 职责 | 诊断能力 |
|-------|------|---------|
| **Metrics Agent** | 指标分析 | 异常检测、相关性分析、趋势预测 |
| **Logs Agent** | 日志分析 | 错误聚类、异常模式识别、上下文提取 |
| **Change Agent** | 变更分析 | 变更关联、变更diff分析、回滚建议 |
| **Topology Agent** | 拓扑分析 | 拓扑发现、影响分析、根因溯源 |
| **Knowledge Agent** | 知识检索 | 案例检索、模式匹配、经验推荐 |

**推理策略**：

1. **变更类问题**：关联变更时间窗口，分析变更diff，判断是否为变更引入
2. **流量上涨问题**：分析QPS趋势，判断是否为容量不足
3. **下游依赖问题**：沿拓扑向下游追溯，定位故障传播路径
4. **服务自身问题**：结合异常堆栈、链路分析、Profiling等深层次可观测能力

**【阶段四】决策输出**

基于分析结论，判断问题范围，决策是否需要进行深层次服务排障流程，触发对应的排障流程，调用Skill。

**决策逻辑**：

```
问题定性结果
    ↓
是否需要深入分析？
    - 是 → 触发专项排障Skill
    │      ├─ CPU问题 → CPU分析Skill
    │      ├─ JVM问题 → GC分析Skill
    │      ├─ 接口问题 → 接口分析Skill
    │      └─ 异常问题 → 异常堆栈分析Skill
    - 否 → 生成总结报告
```

---

**服务分析详细流程**

服务分析会对服务进行系统、应用层的指标进行分析，发现存在问题的指标集合和异常时间范围，通过专家经验，针对多指标关联分析，同时关联异常、变更等数据进行问题定界，进行问题下钻分析。

**流程图**：

```
服务健康问题
    ↓
指标数据采集（"抽血化验"）
    ├─ 系统层指标：CPU使用率、内存占用
    ├─ 运行层指标：JVM状态、异常堆栈
    └─ 接口层指标：RPC成功率、HTTP延迟
    ↓
生成检测报告
    ├─ 时序异常检测
    ├─ 异常波动标记
    └─ 基线偏离分析
    ↓
决策分析
    ├─ 专家经验判断
    │   └─ 规则映射（计算资源不足/内存泄漏）
    └─ RAG知识检索
        ├─ 向量化特征
        ├─ 检索相似案例
        └─ 历史根因分析
    ↓
是否需要深入分析？
    ├─ 否 → 生成总结报告
    └─ 是 → 专项深入检测
            ├─ 链路追踪：调用链分析
            ├─ 日志分析：错误模式识别
            ├─ 线程剖析：锁竞争检测
            └─ 根因定位：模块故障/配置错误/依赖异常
                ↓
            修复建议
```

---

**接口分析详细流程**

接口分析是服务诊断的某个分析路径，会基于指标、链路、日志的可观测数据，进行问题分析。

| 分析维度 | 分析内容 | 输出 |
|---------|---------|------|
| **指标检测** | 针对明确的服务接口，进行指标检测 | 判断是否是单POD、Zone的问题及指标异常时间段 |
| **链路分析** | 通过链路进行错误调用链、慢响应分析 | 关联Span对应的异常堆栈、调用影响链路、Entry等 |
| **日志查询** | 关联特定时间段异常业务日志、框架日志 | 进行问题总结 |

---

**专项分析能力**

| 专项 | 分析内容 | 诊断手段 |
|------|---------|---------|
| **异常堆栈分析** | 异常类型、堆栈信息、影响范围 | 日志聚类、模式识别 |
| **系统负载（CPU）** | CPU使用率、热点代码、进程分析 | Profiling分析、火焰图 |
| **JVM GC分析** | GC次数/耗时、堆内存、线程数 | HeapDump分析、GC日志分析 |

#### 5.2.3 智能处置

根据诊断结果自动或半自动执行修复动作，缩短MTTR。

**处置策略分级**：

| 风险等级 | 典型场景 | 处置方式 | 示例 |
|---------|---------|---------|------|
| **低风险** | 可预测、可回滚、影响面小 | **自动执行** | 服务重启、扩容、清理日志、切换只读实例 |
| **中风险** | 有一定不确定性，但可快速回滚 | **建议+一键执行** | 配置回滚、流量切换、降级开关 |
| **高风险** | 影响面大，或不可逆操作 | **人工审批** | 数据库主从切换、架构变更、数据修复 |

**核心能力**：

1. **Runbook自动化**：将标准运维操作编码为可执行Runbook，与诊断结果联动
2. **安全护栏**：执行前检查影响面、执行中监控关键指标、执行后验证效果、快速回滚机制
3. **人机协作**：高风险操作需人工确认，执行过程可视化可随时中断

#### 5.2.4 复盘优化

自动复盘、知识沉淀、持续优化。

**核心认知**：
- 诊断结果必须被评估：没有反馈的AI无法进化
- 业务采纳率是关键指标：AI建议再好，业务不用等于零
- 持续学习是核心竞争力：每次故障都是学习机会

**评估指标体系**：

| 维度 | 指标 | 计算方式 |
|------|------|---------|
| **模型质量** | 根因定位准确率 | 正确诊断/总诊断数 |
| **业务价值** | AI建议采纳率 | 采纳AI建议次数/总建议数 |
| **运营效率** | 平均诊断时间 | 告警到报告生成时间 |

**持续学习闭环**：

```
诊断结果生成
    ↓
用户打标（正确/错误）
    ↓
数据进入Feedback Pool
    ↓
定期分析Bad Case
    ↓
模型重训练/规则调整
    ↓
A/B测试验证效果
    ↓
全量发布新版本
    ↓
继续收集反馈...
```

**知识沉淀**：
- **故障模式库**：自动提取常见故障模式，形成结构化知识
- **解决方案库**：沉淀有效的修复方案，供后续复用
- **诊断案例集**：典型案例的完整诊断过程，用于培训新人

### 5.3 能力中心层

能力中心层是诊断能力的调度中枢，负责协调多Agent、管理Skill、沉淀知识。

#### 5.3.1 Agent中控

**核心职责**：
- 接收诊断执行层的诊断请求
- 分解任务并分配给各Agent
- 汇总各Agent诊断结果
- 协调Agent间的依赖关系

**Agent调度策略**：

```
诊断请求
    ↓
任务分解
    ├─ 指标分析任务 → Metrics Agent
    ├─ 日志分析任务 → Logs Agent
    ├─ 变更分析任务 → Change Agent
    ├─ 拓扑分析任务 → Topology Agent
    └─ 知识检索任务 → Knowledge Agent
    ↓
并行执行
    ↓
结果汇总
    ↓
综合判定
```

#### 5.3.2 Skill调度机制

**Skill定义**：可复用的诊断能力单元，对应特定的诊断场景。

**Skill分类**：

| Skill类型 | 示例 | 适用场景 |
|----------|------|---------|
| **平台Skill** | CPU诊断Skill、内存诊断Skill、RPC诊断Skill | 基础设施问题 |
| **业务Skill** | 电商订单诊断Skill、推荐CTR诊断Skill、支付成功率诊断Skill | 业务指标问题 |

**Skill调度流程**：

```
问题定性结果
    ↓
Skill匹配（基于问题类型标签）
    ↓
Skill参数填充（服务名、时间窗口等）
    ↓
Skill执行
    ↓
结果返回
```

**Skill使用机制（消费端）**：

| 机制 | 说明 |
|------|------|
| **告警驱动自动化** | Agent自动匹配Skill |
| **全流程反馈** | 上下文/过程/结论卡片 |

**知识类型**：

### 5.4 智能数据层

智能数据层提供统一的、高质量的、可关联的遥测数据，是AI诊断的根基。

#### 5.4.1 原始数据

| 数据类型 | 内容 | 存储系统 |
|---------|------|---------|
| **指标数据（Metrics）** | QPS、延迟、错误率、饱和度（四大黄金指标） | Prometheus / VictoriaMetrics |
| **日志数据（Logs）** | 应用日志、系统日志、错误堆栈 | Elasticsearch / SLS |
| **链路数据（Traces）** | 分布式调用链、Span级别的性能数据 | Jaeger / Tempo |
| **变更数据（Changes）** | 代码发布、配置变更、基础设施变更 | EventStore |
| **拓扑数据（Topology）** | 服务调用关系、依赖图谱 | 图数据库 |

**统一Schema设计**：

| 数据类型 | 统一字段 |
|---------|---------|
| Metrics | service, pod, namespace, node |
| Logs | service, pod, namespace, trace_id |
| Traces | service, pod, namespace, span_id |
| Changes | service, type, time, operator |

#### 5.4.2 U-MODEL统一建模

**核心设计理念**：从"数据驱动"到"建模驱动"

传统方案是被动收集海量数据后反向分析，而我们采用**先建模、后采集**的方式：先对IT世界建模（实体+关系），再针对性采集数据。

**三层关联体系**：

| 层级 | 关联关系 | 说明 |
|------|---------|------|
| **第一层** | 实体 ↔ 实体 | 服务调用关系、主机承载关系、归属关系 |
| **第二层** | 实体 ↔ 数据 | 服务关联到指标、日志、链路、事件 |
| **第三层** | 数据 ↔ 存储 | 数据映射到具体存储系统（Prometheus、ES等） |

**为什么三层都重要？**

- 只有第一层（CMDB）：AI知道"服务之间的依赖关系"
- 有了第二层和第三层（U-MODEL）：AI知道"去哪里查指标、去哪里拉日志"，真正具备端到端的数据访问能力

#### 5.4.3 服务画像

**画像维度**：

| 维度 | 内容 |
|------|------|
| **资源画像** | CPU/内存/磁盘基线、峰值、趋势、异常模式 |
| **性能画像** | 延迟/吞吐/错误、P50/P99、波动特征、SLI达成 |
| **依赖画像** | 强弱依赖拓扑、下游依赖健康度、故障传播路径 |
| **变更画像** | 变更频率、故障关联度、回滚历史 |

**异常特征提取**：

| 特征维度 | 异常模式 | 关联数据 |
|---------|---------|---------|
| 资源瓶颈 | CPU>80%持续5min + 线程池满 | CPU、线程池、GC |
| 内存泄漏 | 堆内存持续增长 + GC频繁 + OOM | JVM内存、GC日志、错误日志 |
| 下游故障 | 接口错误率突增 + 下游依赖超时 | Trace、下游指标、告警事件 |
| 中间件异常 | Redis慢查询激增 + 连接池耗尽 | Redis指标、连接池、错误日志 |
| 变更引发 | 告警时间 ≈ 变更时间 ± 5min | 变更事件、代码Diff、配置变更 |

#### 5.4.4 高阶算法

| 算法类型 | 应用场景 |
|---------|---------|
| **异常检测** | 基于Prophet、Isolation Forest的时序异常检测 |
| **趋势预测** | 预测指标趋势，提前预警 |
| **日志聚类** | 自动聚类相似错误日志，识别异常模式 |
| **知识检索** | 基于向量化的相似案例检索 |

**为什么智能数据底座是根基？**

根据Metoro的深度分析，大多数AI根因分析工具失败，**不是AI算法的问题，而是数据基础的问题**。典型失败原因包括：

1. **缺失遥测数据**：服务未接入Trace/Log/Metrics，AI完全无法诊断
2. **采样丢失**：Trace采样率1%，AI只能看到1%的画面，得出错误根因
3. **Schema不统一**：各信号字段名不一致，AI关联时出错

**XPILOT的数据策略**：

1. **统一Schema**：所有信号使用统一标签（service, pod, namespace, node）
2. **完整数据访问**：AI直接查询存储，无采样、无API限制
3. **自动补全**：对未接入服务，通过eBPF自动采集基础指标
