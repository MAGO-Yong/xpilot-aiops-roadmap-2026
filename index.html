📌 建设背景

现状瓶颈：现有故障应急体系依赖核心指标告警+RCA，仅能覆盖预设场景，无法应对海量多样化告警，研发仍需大量人工排障。

📊 告警覆盖不足

预设场景有限，海量告警无法被及时发现和响应

⏱️ 人工排障成本高

跨指标/链路/日志多维度排查，重复执行相似操作

🎯 业务诉求多样

电商/推荐/支付等业务排障流程各异，平台无法灵活支撑

核心矛盾：基础设施诊断（CPU/RPC/日志）与业务指标诊断（CTR/订单/支付）是两种完全不同的逻辑——前者通用可标准化，后者业务特异需定制。

💡 解决思路

核心理念：将人工排障经验转化为机器可执行的流程，构建"智能发现→深度分析→智能处置"自动化闭环。
🏗️ 平台侧：通用能力沉淀
  • CPU/内存/RPC/日志标准化诊断
  • 开箱即用，全业务自动继承
💼 业务侧：领域知识编排
  • 电商/推荐/支付等业务SOP固化
  • SRE+业务协同孵化，逐步沉淀
ReAct 技术内核：Reasoning（推理）+ Acting（行动）循环，让AI Agent像专家一样动态选择工具、调整策略，而非执行固定流程。

三层时间提效策略

第一层：更快的分类

现状问题：工程师收到告警后，需要手动查询指标、日志、链路、变更等多源数据，平均耗时5-8分钟才能理解问题上下文。
AI方案：自动富化告警上下文，秒级拉取相关遥测数据并生成结构化摘要，包括异常变化点、影响范围、可能原因分析，让工程师1分钟内掌握全局。
预期效果：分类时间从 5-8分钟 → 1分钟

第二层：更快的诊断

现状问题：工程师需要跨系统查询、关联多源数据、人工推理根因，平均耗时30-45分钟才能定位问题。
AI方案：多Agent并行诊断，Metrics Agent、Logs Agent、Change Agent、Topology Agent协同工作，自动关联分析并输出根因判定，将人工排查转化为机器自动分析。
预期效果：诊断时间从 30-45分钟 → 2-5分钟

第三层：更快的修复

现状问题：定位根因后，工程师需要评估修复方案、人工执行操作，平均耗时10分钟才能完成处置。
AI方案：基于根因智能推荐修复方案（回滚、重启、扩容、配置修复），低风险操作自动执行，高风险操作人工确认后执行，实现修复决策到执行的闭环。
预期效果：修复决策时间从 10分钟 → 2分钟

📊 各层详细说明

Layer 1: 数据基础层（Unified Telemetry）

提供统一的、高质量的、可关联的遥测数据，是AI诊断的根基。根据Metoro的深度分析，大多数AI根因分析工具失败，不是AI算法的问题，而是数据基础的问题。
核心能力：
• 指标数据（Metrics）：QPS、延迟、错误率、饱和度（四大黄金指标）
• 日志数据（Logs）：应用日志、系统日志、错误堆栈
• 链路数据（Traces）：分布式调用链、Span级别的性能数据
• 变更数据（Changes）：代码发布、配置变更、基础设施变更
• 拓扑数据（Topology）：服务调用关系、依赖图谱
为什么智能数据底座是根基？

根据Metoro的深度分析，大多数AI根因分析工具失败，不是AI算法的问题，而是数据基础的问题。典型失败原因包括：

• 缺失遥测数据：服务未接入Trace/Log/Metrics，AI完全无法诊断
• 采样丢失：Trace采样率1%，AI只能看到1%的画面，得出错误根因
• Schema不统一：各信号字段名不一致，AI关联时出错
核心设计理念：从"数据驱动"到"建模驱动"

传统方案是被动收集海量数据后反向分析，而我们采用先建模、后采集的方式： 先对IT世界建模（实体+关系），再针对性采集数据。这带来三大优势：统一Schema、天然关联、AI可直接推理。
三层关联体系（参考阿里云U-MODEL）

第一层：实体 ↔ 实体 — 服务调用关系、主机承载关系、归属关系
第二层：实体 ↔ 数据 — 服务关联到指标、日志、链路、事件
第三层：数据 ↔ 存储 — 数据映射到具体存储系统（Prometheus、ES等）

只有具备这三层，LLM Agent才能真正"知道去哪里查指标、去哪里拉日志"。
数据类型 统一字段 说明
Metrics service, pod, namespace, node 指标数据必须包含实体标识
Logs service, pod, namespace, trace_id 日志数据必须包含实体标识
Traces service, pod, namespace, span_id 链路数据必须包含实体标识
Changes service, type, time, operator 变更数据必须关联到服务
U-MODEL vs CMDB

CMDB回答"这台机器是谁的"（静态资产配置）
U-MODEL回答"这个服务出问题会影响谁"（动态可观测图谱）

两者互补，不是替代关系

🥈 Layer 2: 智能发现层（Intelligent Detection）

职责：告警进入诊断流程前的预处理层，解决告警风暴、重复告警、低价值告警等问题
核心功能模块：

功能模块 技术实现 预期效果
告警去重 基于指纹的重复告警识别（相同指标、相同服务、时间窗口内） 减少50%+重复告警
告警关联 基于拓扑关系的告警聚合（上下游服务同时告警时合并） 将多个相关告警合并为1个Incident
动态基线 基于历史数据建立动态阈值（而非静态阈值） 减少30%+误报
告警分级 AI评估业务影响面，自动标记P0/P1/P2 高优告警优先处理
上下文富化 自动关联最近变更、相关服务状态、历史相似告警 为后续诊断提供上下文
处理流程：

原始告警流入
    ↓
[去重] 是否重复告警？
    - 是：标记为重复，不进入诊断流程
    - 否：进入下一步
    ↓
[关联] 是否存在关联告警？
    - 是：合并为Incident，记录关联关系
    - 否：作为独立Incident
    ↓
[分级] 评估业务影响
    - 影响核心业务 → P0/P1
    - 影响非核心 → P2/P3
    ↓
[富化] 添加上下文
    - 最近变更
    - 相关服务状态
    - 历史相似告警
    ↓
输出：富化后的Incident，进入Layer 3诊断
输入
原始遥测数据（Metrics/Logs/Traces）
输出
结构化异常事件、告警通知
与Layer 3的衔接：经过降噪和富化的Incident进入根因诊断层，减少诊断Agent的无效计算。

🥉 Layer 3: 根因诊断层（Root Cause Analysis）

职责：多Agent并行诊断，定位根因
五大诊断Agent：

1. Metrics Agent（指标分析Agent）
• 异常检测：使用Prophet、Isolation Forest检测指标异常
• 相关性分析：关联上下游服务指标，定位影响链路
• 趋势预测：预测指标趋势，提前预警

2. Logs Agent（日志分析Agent）
• 错误聚类：自动聚类相似错误日志
• 异常检测：识别异常日志模式（如OOM、死锁）
• 上下文提取：提取错误堆栈、关键日志行

3. Change Agent（变更分析Agent）
• 变更关联：关联告警时间窗口内的所有变更
• 变更分析：分析变更diff，识别风险点
• 回滚建议：评估回滚可行性和影响

4. Topology Agent（拓扑分析Agent）
• 拓扑发现：自动发现服务调用关系
• 影响分析：分析故障影响面
• 根因溯源：沿拓扑向上游追溯根因

5. Knowledge Agent（知识检索Agent）
• 知识库检索：检索历史相似故障及解决方案
• 模式匹配：匹配已知故障模式
• 经验推荐：推荐基于历史经验的修复方案
核心认知：必须区分告警类型

服务指标告警（RPC延迟、CPU、内存等）和业务指标告警（CTR、订单量、支付成功率等）有本质差异：

• 服务指标：通用性高，诊断SOP标准化，由平台统一建设
• 业务指标：业务线特异，诊断逻辑场景化，需业务共建
双轨制Agent架构设计

采用平台Agent + 业务Agent的双轨模式：

左轨：服务指标诊断Agent —— 平台统一建设，开箱即用
右轨：业务指标诊断Agent —— SRE+业务共建，逐步沉淀

两轨共享Agent调度器和基础设施，但诊断逻辑独立演进。
服务指标诊断Agent：平台统一建设

核心Skill库（开箱即用）：
• CPU诊断Skill：看进程→看线程→看热点代码→火焰图分析
• 内存诊断Skill：HeapDump分析→大对象定位→引用链分析
• RPC诊断Skill：下游依赖分析→网络质量→序列化耗时
• 磁盘/网络诊断Skill：IO热点、连通性、带宽分析

所有业务服务自动继承，无需业务参与。
业务指标诊断Agent：协同孵化模式

孵化流程：SRE识别需求 → 联合梳理SOP → 开发Skill → 灰度验证 → 发布共享

业务Skill示例：
• 电商订单诊断：订单量跌→看流量→看转化→看支付→看履约
• 推荐CTR诊断：CTR跌→看召回→看排序→看曝光→看冷启动
• 支付成功率诊断：成功率跌→看渠道→看风控→看限额→看第三方

相似业务可参考复用，避免重复造轮子。

🏅 Layer 4: 智能处置层（Intelligent Remediation）

职责：根据诊断结果自动或半自动执行修复动作，缩短MTTR
业界实践：业界将自愈能力分为三个层级：Level 1（Auto-Detection，监控告警）、Level 2（Auto-Remediation，智能自动化）、Level 3（全自动自愈）。当前主流AIOps产品（如ServiceNow、BMC Helix）均支持Level 2的自动修复能力。BMC Helix AIOps的实践案例显示，通过自动化修复工作流，MTTR从30-40分钟缩短到5分钟以内。
处置策略分级：

风险等级 典型场景 处置方式 示例
低风险 可预测、可回滚、影响面小 自动执行 服务重启、扩容、清理日志、切换只读实例
中风险 有一定不确定性，但可快速回滚 建议+一键执行 配置回滚、流量切换、降级开关
高风险 影响面大，或不可逆操作 人工审批 数据库主从切换、架构变更、数据修复
核心能力：

1. Runbook自动化
• 将标准运维操作编码为可执行Runbook
• 支持参数化、条件判断、错误处理
• 与诊断结果联动，自动匹配最佳Runbook

2. 安全护栏
• 执行前检查：影响面评估、依赖服务状态
• 执行中监控：实时观察关键指标变化
• 执行后验证：自动验证修复效果
• 快速回滚：失败时自动或一键回滚

3. 人机协作
• 高风险操作需人工确认（通过IM/工单）
• 执行过程可视化，可随时中断
• 执行结果自动记录，用于审计
业界自愈成熟度模型：

• Level 1：Auto-Detection（监控告警）
• Level 2：Auto-Remediation（智能自动化）- 当前目标
• Level 3：全自动自愈
与Layer 3的衔接：Layer 3的诊断报告包含"推荐处置方案"和"风险等级"，Layer 4根据推荐执行相应动作。

🎖️ Layer 5: 复盘与学习层（Postmortem & Learning）

职责：自动复盘、知识沉淀、持续优化
核心认知：

• 诊断结果必须被评估：没有反馈的AI无法进化
• 业务采纳率是关键指标：AI建议再好，业务不用等于零
• 持续学习是核心竞争力：每次故障都是学习机会
业界实践：Salesforce AIOps团队通过三种方式持续改进AI模型：

1. Add runbooks：不断onboard更多Runbook，扩展AI能力覆盖
2. Discover opportunities：月度报告识别覆盖盲区，主动优化
3. Retraining and validation：与AI研究团队持续合作，基于人工输入重新训练和验证模型

研究表明，引入Human-in-the-Loop反馈可以将模型准确率提升20-30%。
反馈收集机制：

反馈类型 收集方式 用途
诊断准确性 诊断报告页面打标（正确/错误/部分正确） 评估模型准确率，识别bad case
建议采纳率 跟踪用户是否采纳AI建议 评估AI建议的实用性和可信度
NPS评分 定期推送NPS调研（0-10分） 评估产品整体满意度
定性反馈 开放文本反馈 收集具体改进建议
评估指标体系：

【模型质量指标】（技术视角）
• 根因定位准确率：正确诊断 / 总诊断数
• 误报率：错误诊断 / 总诊断数
• 漏报率：未识别的真实故障 / 总故障数
• 置信度校准：高置信度诊断的实际准确率

【业务价值指标】（业务视角）
• AI建议采纳率：采纳AI建议的次数 / 总建议数
• 人工诊断替代率：AI直接处理 / 总故障数
• MTTR缩短率：(原MTTR - 现MTTR) / 原MTTR
• NPS评分：用户净推荐值

【运营效率指标】（运营视角）
• 平均诊断时间：从告警到报告生成的时间
• 诊断覆盖率：接入AI诊断的告警 / 总告警数
• 知识沉淀量：新增的故障模式 / 解决方案数量

7.2 数据基础要求

为什么大多数AI RCA失败：
1. 缺失遥测数据 - 服务未接入可观测，AI无法诊断
2. API访问限制 - 采样率1%，AI只能看到1%的图景
3. Schema不一致 - logs用service.name，traces用service，metrics用workload.name
XPILOT的数据策略：
1. 统一Schema：所有信号使用统一标签（service, pod, namespace, node）
2. 完整数据访问：AI直接查询存储，无采样、无API限制
3. 自动补全：对未接入服务，通过eBPF自动采集基础指标
