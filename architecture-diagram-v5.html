📌 建设背景

🔴 现状与瓶颈

现有故障应急体系的局限

我们现有的故障应急体系依赖核心指标告警，关联RCA进行诊断，仅能覆盖预设的业务场景，无法有效应对日常生产中海量的、多样的告警。研发同学仍需投入大量精力进行人工排障。

📊 痛点一：告警覆盖不足

现有体系仅能处理预设的核心指标告警场景，面对日常生产中海量的、多样的告警时显得力不从心。大量潜在风险无法被及时发现和响应。

⏱️ 痛点二：人工排障成本高

研发同学需要针对告警，通过指标、链路、日志等不同维度进行人工排查。每次故障都需要重复执行相似的查询操作，消耗大量时间和精力。

🎯 痛点三：业务诉求多样化

特定业务有特定的排障流程：电商业务查订单、推荐业务查召回、支付业务查渠道。针对不同场景，用户有不同的排障诉求，现有平台无法灵活支撑。

核心矛盾： 业务指标诊断（如CTR跌、订单量跌）和基础设施指标诊断（如CPU高、RPC超时）本质上是两种完全不同的诊断逻辑。

• 基础设施问题：通用性强，可以标准化
• 业务指标问题：业务特异，需要业务知识
🏗️ 基础设施诊断
  • CPU/内存/网络资源
  • RPC调用链分析
  • 日志错误聚类
  • 平台统一建设
💼 业务指标诊断
  • CTR/订单/支付成功率
  • 业务链路追踪
  • 领域知识编排
  • 业务共建沉淀

💡 解决思路：构建"发现-分析"自动化闭环

核心理念：将人工排障经验转化为机器可执行的流程

我们旨在构建"智能发现 → 深度分析"的自动化闭环，并以 ReAct 模式作为技术内核。

🎯 问题发现：如何从"噪音"中找到真问题？

1. 风险感知（智能降噪）
将海量告警作为广泛的风险感知源，进行智能分类、收敛和路由。通过AI自动判断告警优先级和业务影响，过滤无效噪音。

2. 问题定性（快速判断）
对路由来的告警，自动分析关联服务的系统、应用及业务指标，快速判断问题的性质（如时延、错误率、资源瓶颈等），为后续诊断确定方向。

🔍 深度分析：如何像专家一样定位根因？

1. 沉淀通用能力（平台侧）
将观测平台上的通用排障"最佳实践"固化为可复用的分析流程"：
• CPU/内存诊断：看进程→看线程→看热点代码→火焰图分析
• RPC诊断：下游依赖分析→网络质量→序列化耗时
• 日志诊断：错误聚类→异常模式→堆栈分析

2. 编排领域知识（业务侧）
赋能业务方，将他们特有的、强业务语义的排障流程，通过编排原子能力（指标查询、日志查询、SQL查询等）形成可复用的"领域技能"：
ReAct 技术内核

采用 Reasoning（推理）+ Acting（行动） 的循环模式：

Thought（思考）→ Action（行动）→ Observation（观察）→ Thought（再思考）...

让AI Agent像人类专家一样，能够根据当前情况动态选择分析工具、调整诊断策略，而非执行固定流程。
